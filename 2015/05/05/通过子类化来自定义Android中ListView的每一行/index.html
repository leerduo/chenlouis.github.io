通过子类化来自定义Android中ListView的每一行

By louis.chen

发表于 2015-05-05

文章目录

  2. 2. 使用Holder模式的问题

Android程序员在定义ListView中每一行的样式时通常都是创建他们自己的布局然后进行填充数据。当使用这种方式的时候，我们会使用Holder这种模式。但是Holder这种模式很呆板(clumsy and full of builerplate)。我们可以做到更好。在本篇博客中，我们会探索使用一种替代的方式，使用RelativeLayout的子类来封装自定义的工作。

目标

为了达到演示的目的，我们创建了一个典型的ListView的行，每一个Item的父结点是RelativeLayout，包含了一个ImageView和两个TextView。如下图所示：

这个案例的代码在:example project on github
当为ListView或者其他的AdapterView创建自定义的views的时候，通常需求如下：

  • 使用自定义的布局来组织子views
  • 当滑动的时候需要重复利用view recycling
  • 有效的识别和用数据填充子views

    使用Holder模式的问题

    通常使用Holder模式的步骤如下：
  • 为Item创建一个布局
  • 创建一个Holder类，这个类的fields是每一行的child views
  • 创建一个Holder的实例并通过使用findViewById(int)为holder的fields赋值
  • 调用setTag(Object)方法，将Holder的实例作为参数传入
  • 重复利用(并强制类型转化)holder对象，为了重复利用row views
    这种模式的问题在于：
  • Adapter的getView()方法负责太多的任务
  • Holder类是非常呆板的代码
  • view的tag可能需要强制类型转换为holder的类型，这看起来有点缺憾
  • 它破坏了封装，因为adapter/holder必须知道ListView的Item的内部的View对象
    那么通过子类化可以解决这个问题。

    通过子类化自定义

    我们通过使用RelativeLayout的子类来定义布局的根view,这样避免了创建一个泛型的View、一个Holder类、一个Adapter的实现。并把这个类称为：ItemView。Item代表了泛型实体对象，它有三个属性：一个图片的url，一个title,一个描述。

ItemView负责将Item”翻译”成views来渲染屏幕，下面是需要做的两件事:

  • 为每一个Item创建或者复用ItemView的实例
  • 将当前Item的Item和ItemView进行关联

首先，我们通过调用ItemView.inflate(ViewGroup)，一个静态的方法，如果当前没有可以重复利用的ItemView实例，那么就返回一个ItemView实例。然后通过调用setItem(Item)来填充当前的Item。所有的这一切都封装在ItemView中。

inflate(ViewGroup)静态方法提供了使用代码创建一个properly-configured的ItemView，同时也充分利用了xml文件来进行基本的配置。

这个根节点是merge，这意味着在inflation的时候，那个tag的每一个孩子都会被添加当做parent参数的孩子在构造器的inflate(...)方法中。(原文:The root element is a merge tag, which means that during inflation, each of the children of that tag will be added as children of the parent argument passed to the inflate(...) method in the constructor. )
总结上面的，这样做的优点：

  • Adapter的实现很简单
  • ItemView可以使用java代码创建，也可以使用xml文件创建
  • 易于将来的定制和配置
  • 在处理的过程中没有多余的”Holder”
